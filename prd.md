# PRD: MediaReporting

## 1. 문서 목적
MediaReporting 시스템의 기능 범위, 운영 방식, 데이터 흐름, 품질 기준을 정의한다. 이 문서는 개발/운영/협업자가 동일한 기준으로 구현 및 확장할 수 있도록 하는 제품 요구사항 문서(PRD)다.

## 2. 제품 개요
MediaReporting은 기업 관련 뉴스 모니터링 자동화 시스템이다.
핵심 목표는 다음과 같다.
- 기업 키워드 기반으로 뉴스 수집 (Google News RSS, Naver News API)
- 기사 중복 제거 및 리스크 레벨 분류
- Gemini 기반 한국어 요약
- HTML/Markdown 리포트 생성
- Slack/이메일 알림 발송
- 웹 대시보드에서 현황 조회 및 즉시 실행

## 3. 대상 사용자
- PR/홍보팀: 부정 이슈 조기 탐지
- 대외협력/법무: 리스크 기사 모니터링
- 운영 담당자: 파이프라인 상태 확인, 설정 변경

## 4. 핵심 사용자 시나리오
- 시나리오 A: 정기 자동 리포트
  - 03:00 / 09:00 / 15:00 KST에 자동 수집 및 보고서 발행
- 시나리오 B: 이슈 발생 시 즉시 점검
  - 웹에서 즉시 실행 버튼으로 수집/분석/리포트 수동 트리거
- 시나리오 C: 실시간 고위험 알림
  - 스케줄 외 간격 모니터링으로 HIGH/CRITICAL 이슈 알림

## 5. 기능 요구사항

### 5.1 수집
- Google News RSS에서 키워드별 기사 수집
- Naver News API에서 키워드별 기사 수집
- 최근 N시간(`COLLECT_HOURS_BACK`) 기준으로 필터링

### 5.2 데이터 처리
- URL 해시 및 제목 유사도 기반 중복 제거
- 기사별 리스크 레벨 분류: `CRITICAL`, `HIGH`, `MEDIUM`, `LOW`
- 카테고리 분류(정책/법무/제품 등)
- Gemini 모델(`gemini-2.5-flash`) 기반 요약

### 5.3 저장/감사
- SQLite 저장: 기사, 런 이력, 감사 로그, 일간 로그
- 실행별 감사 플래그 기록
- 일간 누적 로그(리스크 분포/상위 기사/키워드 커버리지)

### 5.4 리포팅/알림
- HTML 리포트 생성
- Markdown 리포트 생성
- Slack Webhook 발송
- SMTP 이메일 발송(본문 + HTML 첨부)

### 5.5 웹 대시보드
- 대시보드: 일간 통계, 최근 기사, 최근 리포트
- 피드: 날짜/키워드/리스크 필터
- 설정: 회사명/키워드/리스크/간격 변경
- API:
  - `POST /api/run-now`
  - `POST /api/reset-and-run`
  - `GET /api/run-status`
  - `GET /api/articles`
  - `GET /api/daily/<date>`

## 6. 비기능 요구사항
- 안정성: 외부 API 실패 시 예외 로깅 및 파이프라인 실패 기록
- 관측성: 로그 파일 로테이션, 실행 이력 저장
- 확장성: 모듈 구조로 수집기/알림기 추가 가능
- 배포성: 로컬 실행 + Vercel 웹 배포 동시 지원

## 7. 아키텍처 개요
- 엔트리포인트: `main.py` (스케줄러/모니터)
- 웹: `web/app.py` (Flask)
- 스케줄 오케스트레이션: `scheduler/jobs.py`
- 저장소: `storage/database.py` (SQLite)
- 외부 연동:
  - Naver API
  - Google News RSS
  - Gemini API
  - Slack Webhook
  - SMTP

## 8. 데이터 모델(개요)
- `articles`: 기사 본문 메타/요약/리스크/알림 여부
- `report_runs`: 실행 단위 메타/결과물 경로
- `audit_log`: 감사 플래그
- `daily_monitoring_log`: 일간 누적 관측치

## 9. 운영 정책
- 정기 실행 시간: KST 03:00 / 09:00 / 15:00
- 모니터링 간격: 기본 300초(환경변수 조정)
- 고위험 기준: 키워드 기반 규칙 + 분류기

## 10. 성공 지표(KPI)
- 고위험 기사 탐지 누락률 감소
- 정기 리포트 발행 성공률
- 수집 대비 중복 제거 효율
- 알림 발송 성공률(Slack/Email)

## 11. 제약/리스크
- 외부 API 쿼터 및 일시 장애 영향
- Naver API 키 미설정 시 수집 편향
- SMTP 보안 정책(앱 비밀번호) 미충족 시 이메일 실패

## 12. 향후 확장 범위
- 다국어/다국가 소스 확장
- 룰 기반 + 모델 기반 하이브리드 분류 고도화
- 관리자 인증 및 멀티테넌트 회사 분리
- 배치/스트리밍 모드 전환
